## Core Principles

### Everything is Really Just Search
GET methods use search parameters for composability, allowing you to combine filters and find exactly the data you need.

### Always an Array
All API methods accept and return arrays, enabling efficient batch operations for high-volume integrations.

### Summary vs Detail
Control response size and processing time using the `list_type` parameter:
- `summary`: Basic information for listings and overviews
- `detail`: Complete information for specific operations

### Robust Error Handling
Bulk operations are non-blocking with individual success/failure status for each item, ensuring partial failures don't block entire batches.

## Data Formats

### Date and Time
All timestamps use **RFC3339** format for consistency:
```
2024-01-15T10:30:00Z
```

### Pagination
Standard pagination parameters across all list endpoints:

| Parameter | Type | Description | Default |
|-----------|------|-------------|---------|
| `limit` | integer | Number of results [1..100] | 30 |
| `offset` | integer | Pagination offset | 0 |

### Response Format
All responses follow a consistent structure:

```json
{
  "limit": 30,
  "offset": 0,
  "list_type": "detail",
  "data": [
    // Array of results
  ]
}
```

## Rate Limits and Best Practices

<Note>
All API endpoints require HTTPS with a valid JWT Bearer token. Rate limits apply to ensure system stability.
</Note>

### Recommended Patterns

1. **Use Summary Lists**: Start with `list_type=summary` for overviews
2. **Batch Operations**: Group related operations into single requests
3. **Efficient Filtering**: Use specific search parameters to reduce response sizes
4. **Monitor Callbacks**: Implement webhook handlers for real-time updates

## Error Handling

The API uses standard HTTP status codes and provides detailed error information:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Required field missing",
    "details": {
      "field": "shipping_address_name",
      "value": null
    }
  }
}
```
