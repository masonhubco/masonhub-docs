## Authentication

All API endpoints require HTTPS with a valid JWT Bearer token. Generate tokens through the [Token Management](/api-reference/tokens/generate-new-bearer-token) endpoints.

<Note>
Bearer tokens begin with the `eyJ` prefix and must be included in the Authorization header of all requests. Keep your bearer tokens secret.
</Note>

### Using Your Token

Include the bearer token in the Authorization header:

```bash
curl -X GET "https://app.masonhub.co/{account}/api/v1/orders" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

## API Design Principles

The MasonHub API follows consistent design principles for a predictable and powerful developer experience:

<CardGroup cols={2}>
  <Card
    title="Search-First Design"
    icon="magnifying-glass"
  >
    GET methods use search parameters for composability and flexible data retrieval. Everything is really just search.
  </Card>
  <Card
    title="Batch Operations"
    icon="layer-group"
  >
    All methods accept and return arrays for efficient bulk operations. Process multiple items in single API calls.
  </Card>
  <Card
    title="Flexible Responses"
    icon="toggle-on"
  >
    Use `list_type=detail|summary` to control response detail levels and optimize data transfer.
  </Card>
  <Card
    title="RESTful Design"
    icon="code"
  >
    Standard GET, POST, PUT, DELETE operations. No PATCH to maintain simplicity.
  </Card>
  <Card
    title="Robust Error Handling"
    icon="shield-check"
  >
    Bulk operations are non-blocking with individual success/failure status for each item.
  </Card>
  <Card
    title="Consistent Formatting"
    icon="calendar"
  >
    RFC3339 date format used throughout for consistency and international support.
  </Card>

## Data Formats

### Date and Time

All timestamps use **RFC3339** format for consistency and timezone support:

```json
{
  "created_at": "2024-01-15T10:30:00Z",
  "updated_at": "2024-01-15T14:45:30Z"
}
```

### Pagination

Standard pagination parameters across all list endpoints:

<ParamField query="limit" type="integer" default="30">
  Number of results to return. Range: 1-100.
</ParamField>

<ParamField query="offset" type="integer" default="0">
  Pagination offset for retrieving subsequent pages of results.
</ParamField>

<ParamField query="list_type" type="string" default="detail">
  Response detail level. Options:
  - `detail` - Full information including all fields
  - `summary` - Condensed view with essential fields only
</ParamField>

### Response Structure

All list responses follow a consistent structure:

```json
{
  "limit": 30,
  "offset": 0,
  "list_type": "detail",
  "data": [
    {
      // Array of result objects
    }
  ]
}
```

### Batch Operation Responses

Bulk operations return detailed status for each item:

```json
{
  "records_submitted": 5,
  "records_processed": 5,
  "records_failed": 1,
  "records_succeeded": 4,
  "results": [
    {
      "customer_identifier": "item-001",
      "status": "success"
    },
    {
      "customer_identifier": "item-002",
      "status": "failure",
      "error_message": "SKU not found in catalog"
    }
  ]
}
```

Ready to integrate with MasonHub? Follow these steps:

<Steps>
  <Step title="Generate API Token">
    Create your JWT bearer token using the [Token Management](/api-reference/tokens/generate-new-bearer-token) endpoint
  </Step>
  <Step title="Test in Sandbox">
    Use the sandbox environment to test your integration without affecting production data
  </Step>
  <Step title="Create Catalog">
    Set up your [SKUs](/api-reference/catalog/create-skus) and product catalog
  </Step>
  <Step title="Configure Callbacks">
    Set up [webhooks](/api-reference/callbacks/create-callback-urls) for real-time event notifications
  </Step>
  <Step title="Test with DataFactory">
    Use [DataFactory](/api-reference/data-factory/create-available) to simulate operations and validate your integration
  </Step>
  <Step title="Go Live">
    Move to production with confidence after thorough testing
  </Step>
</Steps>
