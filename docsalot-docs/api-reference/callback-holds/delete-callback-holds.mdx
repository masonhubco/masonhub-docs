---
title: 'Delete Callback Holds'
description: 'Remove holds to resume callback delivery for specific message types'
api: 'DELETE /callback_holds'
---

Remove callback holds to resume delivery of specific message types to your webhook endpoints. After removing holds, new events will be delivered according to your callback configuration.

<Note>
Removing holds only affects future events. Events that occurred while holds were active are not retroactively delivered.
</Note>

## Request Body

The request body accepts an array of message type strings to remove from hold.

<ParamField body="message_types" type="string[]" required>
  Array of message types to remove from hold. Common types include:
  - `skuInventoryChange` - Inventory level changes
  - `orderEvent` - Order status and lifecycle events
  - `shipmentEvent` - Shipment tracking updates
  - `inboundReceived` - Inbound shipment receipts
</ParamField>

<RequestExample>
```bash cURL
curl -X DELETE "https://app.masonhub.co/{account}/api/v1/callback_holds" \
  -H "Authorization: Bearer your_jwt_token" \
  -H "Content-Type: application/json" \
  -d '[
    "skuInventoryChange",
    "orderEvent"
  ]'
```

```python Python
import requests
import json

url = "https://app.masonhub.co/{account}/api/v1/callback_holds"
headers = {
    "Authorization": "Bearer your_jwt_token",
    "Content-Type": "application/json"
}

data = [
    "skuInventoryChange",
    "orderEvent"
]

response = requests.delete(url, headers=headers, json=data)
result = response.json()
```

## Response Fields

<ResponseField name="status" type="string">
  Status of the operation. Returns `success` when holds are removed.
</ResponseField>

<ResponseField name="holds_removed" type="string[]">
  Array of message types that were successfully removed from hold.
</ResponseField>

<ResponseField name="remaining_holds" type="string[]">
  Array of message types that still have active holds after this operation.
</ResponseField>

## Behavior

### Idempotent Operations
If you attempt to remove a hold that doesn't exist, the API will return a 404 error but will still process other valid hold removals in the request.

### Multiple Message Types
You can remove holds from multiple message types in a single request for efficiency.

### Immediate Effect

<Info>
Once holds are removed, callback delivery resumes immediately for new events. The system does not retroactively deliver events that occurred during the hold period.
</Info>

## Partial Removal Handling

If some holds cannot be removed (e.g., not found), the API will still remove other valid holds:

```json
{
  "status": "partial_success",
  "holds_removed": ["skuInventoryChange"],
  "remaining_holds": ["orderEvent"],
  "errors": [
    {
      "message_type": "shipmentEvent",
      "error": "Hold not found"
    }
  ]
}
```

## Example Workflow

```python
# Complete maintenance tasks
complete_system_maintenance()

# Verify endpoints are ready
if verify_webhook_endpoints():
    # Remove holds to resume callbacks
    hold_types = ["skuInventoryChange", "orderEvent"]
    response = requests.delete(
        f"{base_url}/callback_holds",
        headers=headers,
        json=hold_types
    )

    # Verify removal
    current_holds = requests.get(
        f"{base_url}/callback_holds",
        headers=headers
    ).json()

    if len(current_holds["holds"]) == 0:
        print("All holds removed successfully")
    else:
        print(f"Remaining holds: {current_holds['holds']}")
```

## What Happens After Removal

<Steps>
  <Step title="Immediate Resume">
    Callback delivery resumes immediately for the specified message types.
  </Step>
  <Step title="New Events Only">
    Only events occurring after hold removal are delivered.
  </Step>
  <Step title="Normal Retry Logic">
    Standard retry and timeout policies apply to all subsequent deliveries.
  </Step>
  <Step title="No Backfill">
    Events missed during the hold period are not retroactively sent.
  </Step>
</Steps>

<Note>
If you need to retrieve events that occurred during a hold period, consider using the relevant GET endpoints (e.g., GET /inventory, GET /orders) to fetch current state.
</Note>
