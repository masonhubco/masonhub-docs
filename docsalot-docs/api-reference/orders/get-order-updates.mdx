---
title: 'Get Order Updates'
description: 'Retrieve status and details of order update requests'
api: 'GET /order_update_requests'
---

Retrieve the status and details of previously submitted order update requests. Use this endpoint to monitor the asynchronous processing of order modifications and verify successful updates.

<Note>
Order update requests are processed asynchronously. Poll this endpoint or configure webhooks to track completion status.
</Note>

## Query Parameters

Filter and retrieve update requests with these parameters:

<ParamField query="id" type="string">
  Filter by update request system ID (UUID). Returns specific update request.
</ParamField>

<ParamField query="cid" type="string">
  Filter by customer identifier. Returns all update requests for a specific order.
</ParamField>

<ParamField query="status" type="string">
  Filter by request status:
  - `pending` - Awaiting processing
  - `processing` - Currently being processed
  - `success` - Update completed successfully
  - `failed` - Update failed with errors
  - `partial` - Some updates applied, others failed
</ParamField>

<ParamField query="limit" type="integer" default="100">
  Maximum number of records to return (1-1000).
</ParamField>

<ParamField query="offset" type="integer" default="0">
  Number of records to skip for pagination.
</ParamField>

<ParamField query="created_after" type="string">
  Filter requests created after this timestamp (RFC3339 format).
</ParamField>

<ParamField query="created_before" type="string">
  Filter requests created before this timestamp (RFC3339 format).
</ParamField>

<ParamField query="resolved_after" type="string">
  Filter requests resolved after this timestamp (RFC3339 format).
</ParamField>

<ParamField query="resolved_before" type="string">
  Filter requests resolved before this timestamp (RFC3339 format).
</ParamField>

## Response Fields

<ResponseField name="system_id" type="string">
  Unique identifier for the update request
</ResponseField>

<ResponseField name="customer_identifier" type="string">
  Your order reference ID
</ResponseField>

<ResponseField name="status" type="string">
  Current status of the update request:
  - `pending` - Queued for processing
  - `processing` - Currently being applied
  - `success` - All updates applied successfully
  - `failed` - Updates could not be applied
  - `partial` - Some updates applied, others failed
</ResponseField>

<ResponseField name="submitted_at" type="string">
  Timestamp when update request was submitted
</ResponseField>

<ResponseField name="resolved_at" type="string">
  Timestamp when processing completed (null if pending)
</ResponseField>

<ResponseField name="updates_applied" type="object">
  Details of successfully applied changes with old/new values
</ResponseField>

<ResponseField name="updates_failed" type="object">
  Details of failed updates with error reasons
</ResponseField>

<ResponseField name="error" type="object">
  Error details if entire request failed
</ResponseField>

<ResponseField name="processing_time_ms" type="integer">
  Time taken to process the request in milliseconds
</ResponseField>

<ResponseField name="order_status" type="string">
  Current status of the order after update
</ResponseField>

## Update Request Lifecycle

<Steps>
  <Step title="pending">
    Request received and queued for processing
  </Step>
  <Step title="processing">
    System actively applying updates to the order
  </Step>
  <Step title="Resolution">
    Final status determined:
    - `success` - All changes applied
    - `failed` - No changes applied
    - `partial` - Some changes applied
  </Step>
</Steps>

## Polling Strategy

Implement efficient polling to monitor update status:

```python
import time
import requests

def wait_for_update_completion(update_id, headers, timeout=300):
    """
    Poll for update completion with exponential backoff
    """
    url = f"https://app.masonhub.co/{{account}}/api/v1/order_update_requests"
    start_time = time.time()
    poll_interval = 5  # Start with 5 seconds

    while time.time() - start_time < timeout:
        response = requests.get(
            url,
            headers=headers,
            params={"id": update_id}
        )

        data = response.json()
        if data["update_requests"]:
            status = data["update_requests"][0]["status"]

            if status in ["success", "failed", "partial"]:
                return data["update_requests"][0]

        time.sleep(poll_interval)
        # Exponential backoff with max 60 seconds
        poll_interval = min(poll_interval * 1.5, 60)

    raise TimeoutError(f"Update {update_id} did not complete within {timeout} seconds")
```

## Best Practices

<CardGroup cols={2}>
  <Card title="Use Webhooks" icon="webhook">
    Configure [orderUpdateResolution](/api-reference/callback-events/order-update-resolutions) callbacks instead of polling
  </Card>
  <Card title="Implement Backoff" icon="clock">
    Use exponential backoff when polling to reduce API load
  </Card>
  <Card title="Handle Partials" icon="circle-half-stroke">
    Check both `updates_applied` and `updates_failed` for partial successes
  </Card>
  <Card title="Track Request IDs" icon="fingerprint">
    Store system_id from submission for reliable status tracking
  </Card>
</CardGroup>

<Warning>
Update requests older than 30 days may be archived and unavailable through this endpoint. Store important update history in your own system.
</Warning>

## Webhook Alternative

For real-time updates without polling, configure webhooks:

```bash
POST /callbacks
{
  "callback_url": "https://your-api.com/webhooks/order-updates",
  "event_type": "orderUpdateResolution"
}
```

You'll receive notifications when update requests complete:

```json
{
  "message_type": "orderUpdateResolution",
  "data": [
    {
      "update_request_id": "0b744e29-b668-4486-85dd-82528b5da0dd",
      "customer_identifier": "129374",
      "status": "success",
      "resolved_at": "2024-01-17T10:00:45Z"
    }
  ]
}
```

<Tip>
For high-volume operations, use webhooks instead of polling. This reduces API calls and provides faster notification of update completions.
</Tip>
