---
title: 'Delete Inbound Shipments'
description: 'Soft delete inbound shipments by customer identifier'
api: 'DELETE /inbound_shipments'
---

Soft delete inbound shipments that are no longer needed. This endpoint performs a logical deletion, maintaining records for audit purposes.

<Warning>
Only shipments with `open` status can be deleted. Shipments that are already on dock or being received cannot be removed.
</Warning>

<Note>
This is a soft delete operation. Shipments are marked as deleted but remain in the system for historical tracking and reconciliation.
</Note>

## Request Body Schema

Send an array of objects containing the customer identifiers of shipments to delete.

<ParamField body="customer_identifier" type="string" required>
  Unique shipment identifier that matches the existing shipment you want to delete.
</ParamField>

<RequestExample>
```bash cURL
curl -X DELETE "https://app.masonhub.co/{account}/api/v1/inbound_shipments" \
  -H "Authorization: Bearer your_jwt_token" \
  -H "Content-Type: application/json" \
  -d '[
    {
      "customer_identifier": "shipment234523"
    },
    {
      "customer_identifier": "shipment234524"
    }
  ]'
```

```python Python
import requests
import json

url = "https://app.masonhub.co/{account}/api/v1/inbound_shipments"
headers = {
    "Authorization": "Bearer your_jwt_token",
    "Content-Type": "application/json"
}
payload = [
    {
        "customer_identifier": "shipment234523"
    },
    {
        "customer_identifier": "shipment234524"
    }
]

response = requests.delete(url, headers=headers, data=json.dumps(payload))
data = response.json()
```

## Delete Behavior

When you delete an inbound shipment:

<Steps>
  <Step title="Status Validation">
    System checks that the shipment is in `open` status
  </Step>
  <Step title="Soft Delete">
    Shipment is marked as deleted but remains in the system
  </Step>
  <Step title="Inventory Adjustment">
    Expected inventory is reduced for all line item SKUs
  </Step>
  <Step title="Callback Triggered">
    **skuInventoryChange** callbacks are sent for affected SKUs
  </Step>
</Steps>

## When to Delete Shipments

<AccordionGroup>
  <Accordion title="Cancelled Orders">
    Delete shipments when purchase orders are cancelled before the inventory ships.
  </Accordion>

  <Accordion title="Duplicate Entries">
    Remove duplicate shipment entries created by mistake.
  </Accordion>

  <Accordion title="Changed Plans">
    Delete shipments when business plans change and inventory won't be sent.
  </Accordion>

  <Accordion title="Data Correction">
    Remove incorrect shipments and recreate them with the correct information.
  </Accordion>
</AccordionGroup>

## Batch Operations

You can delete multiple shipments in a single request. The system processes each deletion independently:

- **Full Success**: All shipments deleted successfully (HTTP 200)
- **Partial Success**: Some deletions succeeded, others failed (HTTP 207)
- **Full Failure**: All deletions failed (HTTP 400)

<Tip>
When batch deleting, check the response carefully. The API returns detailed information about which shipments were successfully deleted and which failed, including specific error reasons.
</Tip>

## Status Restrictions

<Warning>
Shipments can only be deleted when in **open** status. Once a shipment moves to:
- **onDock**: Shipment has arrived - contact MasonHub support
- **receivingStarted**: Processing has begun - deletion not allowed
- **receivingComplete**: Processing complete - shipment is closed
</Warning>

## Inventory Impact

Deleting an inbound shipment immediately affects expected inventory:

- Expected quantities are reduced for all line items
- Available and allocated inventory remain unchanged
- **skuInventoryChange** callbacks notify you of the changes
- The system maintains accurate inventory projections

<Info>
If you accidentally delete a shipment, you can recreate it using the same `customer_identifier` as long as no new shipment with that identifier has been created.
</Info>
